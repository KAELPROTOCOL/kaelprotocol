## Gaps The Author Should Know Before The Audit

- The new `reorg_rollback_removes_confirmed_counterparty_leg_without_secret_leak`
  test simulates local rollback, but an auditor will likely still ask for
  competing fork behavior and public-chain reorg calibration.
- Confirmation depth is global, not per chain. The binaries default to `3`, but
  local/private scripts force `1`. This is fine for Anvil reproduction and weak
  for public chains. Public calibration is documented as a blocker, not solved.
- Chain observation trusts one RPC per chain. A dishonest or lagging RPC can
  still make a leg look Safe. The code architecture has an interface, but quorum
  or trust-minimized verification is not implemented.
- Restart/recovery is only lightly covered by "re-derive from chain observation"
  logic. There is no durable journal, no crash-at-each-step replay suite, and no
  idempotent resend policy.
- Fee liveness is not production-grade. There is no fee bumping, replacement,
  stuck transaction handling, mempool monitoring, or deadline-aware gas policy.
- The private testnet proves the happy path and expected failures on Anvil, not
  adversarial public-chain network conditions.
- ERC-20 coverage includes a mock and EOA-token rejection, but non-standard
  tokens remain a real audit target: fee-on-transfer, rebasing, blacklists,
  return-value quirks, permit-like behavior, and allowance race semantics are
  not exhaustively covered.
- Settlement canonical HTLC binding is checked by preflight/broadcast and some
  behavior tests, but auditors may want direct constructor/property assertions
  and deployment-manifest checks.
- "Preflight sends no transactions" is true by code inspection and printed by
  the command, but there is no explicit transaction-count test proving it.
- Log/secret hygiene has a focused Debug redaction test and regex scans, but not
  an exhaustive runtime log-capture test across all scripts and failure paths.
- Portuguese comments found during this snapshot were translated in the
  follow-up pre-audit gap pass. Future reviews should keep code and docs in
  English.
- The allowlist includes public testnets such as Sepolia/Holesky/Base Sepolia.
  That is not mainnet, but it is broader than local-only. If "no public value
  testnet" is a hard launch gate, auditors will ask how operators prevent
  accidental use of public testnet assets or public RPCs.
- The orderbook is still in-memory for MVP. This is compatible with non-custody,
  but production behavior around restart, replay, cancellation, and maker order
  lifecycle is not complete.
- Public p2p transport is not built. The docs correctly mark it as out of
  scope/deferred, but a full product audit will need authentication, replay
  resistance, peer identity, message persistence, and denial-of-service review.
- No native Bitcoin implementation exists. SHA-256 hashlocks are compatible with
  future Bitcoin work, but that does not prove Bitcoin-side safety.
- The secrets scan is heuristic. It found no real private key, but any 32-byte
  value can be a private key. A professional audit should run dedicated secret
  scanning tools and review history with higher confidence.
- Some evidence is runner-based rather than unit-test-based. The private runner
  is useful, but auditors may ask for more narrow tests that fail with clearer
  causality.
- The repo is ready for professional audit review of the local/private
  mainnet-readiness package, but it is not production-ready and not mainnet-ready.
  External audit and remediation remain mandatory before any real funds.
